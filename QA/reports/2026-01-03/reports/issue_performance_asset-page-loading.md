## 문제 설명

asset 페이지(/assets) 로딩 시 컴퓨터가 멈출 정도로 심각한 성능 문제가 발생합니다. 페이지가 로드되면서 브라우저가 응답하지 않고, 사용자 경험이 매우 저하됩니다.

## 재현 단계

1. 개발 서버 실행 (`npm run dev`)
2. 브라우저에서 `http://localhost:5173/assets` 접속
3. 페이지 로딩 중 브라우저 응답 지연 및 멈춤 현상 관찰

## 예상 동작

- asset 페이지가 정상적으로 로드되어야 함
- 페이지 로딩 중에도 브라우저가 응답해야 함
- 필터링, 정렬, 검색 등 기능이 부드럽게 작동해야 함

## 실제 동작

- asset 페이지 로딩 시 브라우저가 멈추는 현상 발생
- 페이지 로딩 시간이 매우 길어짐
- 사용자 인터랙션이 불가능한 상태가 지속됨

## 원인 분석

성능 병목 지점:

1. **필터링 로직 비효율**: 매번 모든 row를 순회하면서 복잡한 계산 수행
   - 검색어 매칭 시 매번 문자열 배열을 생성하고 조인
   - severity, vehicleHealth 계산이 매번 반복됨

2. **중복 계산**: 
   - `sortAccessor`와 `filterAccessor`에서 동일한 계산 반복
   - 셀 렌더링 시 `diagnosticCodes` 배열을 매번 순회하여 severity 계산

3. **데이터 변환 비효율**:
   - `withManagementStage` 호출 후에도 추가 계산이 반복됨
   - 검색 가능한 필드들을 매번 조인하여 필터링

## 해결 방안

1. **데이터 전처리 최적화**: rows 로드 시 미리 계산된 값 저장
   - `_searchText`: 검색 가능한 필드들을 미리 조인
   - `_severityComputed`: 진단 코드의 최대 심각도 미리 계산
   - `_vehicleHealthComputed`: 차량 상태 레이블 미리 계산

2. **필터링 로직 최적화**: 미리 계산된 값 활용
   - 검색어 매칭 시 `_searchText` 사용
   - 필터링이 필요 없을 때 조기 반환

3. **정렬/필터링 최적화**: 미리 계산된 값 우선 사용
   - `sortAccessor`와 `filterAccessor`에서 캐시된 값 사용
   - 중복 계산 완전 제거

4. **셀 렌더링 최적화**: 미리 계산된 값 사용
   - severity, vehicleHealth 셀 렌더링 시 중복 계산 제거
